### The Five Questions

**Q1 — What data does this feature display?**
None displayed. Data created/stored:
- `documents`: `_id`, `tenant_id`, `filename`, `mime_type`, `file_size`, `sha256_hash`, `s3_key`, `uploaded_by`, `uploaded_at`, `source_event`, `metadata` (title, author, page_count, language), `status`, `chunk_count`.
- `chunks`: `_id`, `document_id`, `tenant_id`, `chunk_index`, `text`, `token_count`, `start_offset`, `end_offset`, `embedding?`, `metadata` (page_number, section_heading).
- Search response: matched chunks + parent doc refs, `tenant_id`-scoped.

**Q2 — Where does that data live today?**
- `documents` collection: does not exist.
- `chunks` collection: does not exist.
- Raw S3 uploads: LIVE.
- Ingestion / chunking / hashing / embedding: does not exist.

**Q3 — Is there a working endpoint today?**
No. `POST /api/v1/documents/search` is not present in any backend controller.

**Q4 — Can we enter this data ourselves?**
No. Data is *derived* from S3 upload events by a pipeline. Manual entry is not the feature — dedup + chunking + indexing is.

**Q5 — What is the Latha task?**
The entire ticket. Specifically:
1. Create `documents` collection + Mongoose schema.
2. Create `chunks` collection + schema, indexed on `(tenant_id, document_id)`.
3. Add SHA-256 uniqueness constraint per `tenant_id` for dedup.
4. Build ingestion pipeline: subscribe to S3 upload event → metadata extract → SHA-256 → dedup check → chunk → write `chunks` → mark document `processed`.
5. Build `POST /api/v1/documents/search` controller in `apps/backend-nest/src/modules/documents-module/` with mandatory `tenant_id` scoping (use existing scope guard).
6. Decide: keyword (Mongo text index) vs semantic (Atlas vector + embedding model).
7. E2E test: upload twice → second returns existing `document_id`, no duplicate chunks.
