### ISSUE 1 — AGENT exposed to internal sidebar item (CRITICAL — locked-decision violation)
- **Element:** `SideMenu` → `'Manage User Accounts'`, `permissionLevel`
  `libs/shared/models/sidemenu.ts:446-520`
- **Symptom:** `permissionLevel` includes `UserTypes.AGENT` and `UserTypes.ORG_AGENT`. Agent sees a top-level "Manage User Accounts" with leaf "Service Users".
- **Locked decision violated:** AGENT must never see internal portal sections (per ticket `## Why` + `## Acceptance criteria`).
- **Category:** 4 (integration/runtime) · **Risk:** HIGH (audit blocker)
- **Test after correction:** localhost as AGENT → "Manage User Accounts" must be absent.

### ISSUE 2 — AGENT sees internal Applications tree (HIGH)
- **Element:** `SideMenu` → `'Applications'`, `permissionLevel`
  `libs/shared/models/sidemenu.ts:282-352`
- **Symptom:** AGENT and ORG_AGENT in parent `permissionLevel`. The 11 child paths point to internal collection routes (`/${CollectionNames.ApartmentApplications}` etc.), not the agent-scoped `/service/...` variant used by the "Services (agent variant)" tree.
- **Concern:** Either backend filters scope this per-agent (intentional) or this is a parallel leak to Issue 1.
- **Category:** 4 · **Risk:** HIGH (potential cross-tenant data leak)
- **Question Q2 below.**
