## 5-Question Audit

**Q1 — What data does this feature display?**

The "data" for this feature is permission/menu configuration, not user records:

- Menu node fields: `title`, `icon`, `path`, `permissionLevel[]`, `requiredRole`, `children[]`
- User type roles: `SUPER_ADMIN`, `ORG_ADMIN`, `ORG_OWNER`, `MANAGER`, `TEAM_LEAD`, `STAFF`, `AGENT`, `ORG_AGENT`, `USER`
- Role-name gating: `SidebarRole.CallCenter`, `SidebarRole.SalesSetup`, `SidebarRole.Finance` (etc.)
- Collection-permission gating from `user.roles[].permissions[].collection` + `actions`
- Org-service gating from `user.organization[0].orgServices[].service`

**Q2 — Where does that data live today?**

Option A — **TypeScript constants in the codebase**, all confirmed present:

| Artefact | Location | Status |
|---|---|---|
| `SideMenu` registry | `libs/shared/models/sidemenu.ts` (528 lines) | LIVE |
| `exclusivePermission` filter | `libs/shared/functions/permission.helper.ts` (245 lines) | LIVE |
| `UserTypes` enum | `libs/shared/constants/user.types.ts` | LIVE |
| `SidebarRole` enum | `libs/shared/constants/lead.constants.ts` | LIVE (Phase 1 DONE) |
| `CollectionNames` | `libs/shared/constants/collections.ts` | LIVE |
| `user.roles[].permissions[]` | Mongo `users` collection | LIVE |

No Google Sheets dependency. No external source. No missing collection.

**Q3 — Is there a working endpoint today?**

Not applicable for the sidebar config itself — sidebar resolution is **client-side**, computed from the imported `SideMenu` constant filtered through `exclusivePermission(SideMenu, user, referredUser, agentUser)`. The only backend dependency is the existing user-with-roles fetch, which is already LIVE.

**Q4 — Can we enter this data ourselves?**

The question does not apply in the standard sense — this is configuration code, not user-entered data. Shon edits `sidemenu.ts` and `permission.helper.ts` directly. No data-entry UI needed. No empty state needed (every role always has at least one menu item).

**Q5 — What is the Latha task?**

**No data task for Latha.** The portal-side change set produced by Mode 3 will be handed to Latha for review/merge via `/latha-handover` (one capability per commit), but there is no collection to create, no endpoint to add, and no data import.
