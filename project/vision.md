## THE 369 ASSESSMENT SYSTEM

Every document that enters the portal is assessed
by AI using a defined checklist for that document
type. The result is a transparent, trusted score.

Document types with defined checklists:
  Passport
  Bank statement
  Academic certificate
  Language certificate (IELTS, TOEFL, PTE etc)
  Motivation letter
  Visa / Residence permit
  Blocked account confirmation

Each check either PASSES, FAILS or shows
CANNOT VERIFY with the specific reason.

Five verdicts:
  APPROVED — all required checks passed
  APPROVED WITH NOTE — passed with advisory note
  NEEDS REVIEW — some checks could not complete
  REJECTED — one or more checks failed
  CANNOT PROCESS — document unreadable

No black boxes. Every check is transparent.
Staff read the checklist. Students understand
exactly what failed and what to resubmit.
Universities trust the verification is rigorous.

This is a competitive advantage.
No competitor offers this level of transparency.

## BUILD PRINCIPLES (non-negotiable)

1. Tenant-aware from day one
   Every feature configurable per organisation
   No hardcoded DeAssists-specific logic
   What works for DeAssists must work for BCBT
   and the next 50 universities without changes

2. AI-hookable at every action
   Every state change fires an event
   AI agents act without human involvement
   Design for automation from day one

3. Harness-first development
   Every feature is a ticket
   EAGLE Harness builds from the ticket
   Test Harness verifies automatically
   One task, one commit, one Latha review
   No feature ships without harness approval

4. Constants before components
   Enums exist before code references them
   Constants file is a hard gate
   No magic strings anywhere

5. No hardcoded business values
   Statuses, queues, roles, services,
   fees, rules — all configurable per tenant

6. Four-layer API chain — no exceptions
   Component then Hook then Core Hook
   then Axios then Backend

7. Operator today, platform tomorrow
   Every decision supports both simultaneously

8. AI-first design
   If a human does it repetitively today,
   design it so AI can do it tomorrow
   without a rebuild

9. Transparent over clever
   No black boxes — not in AI decisions, not in
   code, not in business logic
   Every check visible. Every reason explainable.

10. Architecture before prototype
    For every new layer:
    Understand then Architecture then Prototype
    then UI/UX review then Finalise then Build
